>_EXECUTIVE SUMMARY

zemacs-gui is a thin Tauri v2 desktop host (app/src-tauri) that runs the external zemacs editor — a Helix fork — inside an embedded PTY terminal, behind the shared zgui-core app baseline. The Rust host is small by design (~1140 LOC across 7 modules): it resolves the bundled zemacs/stryke sidecars, registers the PTY terminal (shared zpwr-embed-terminal crate), and exposes MacVim-style helpers (file-open intake, read-only directory listing, window chrome) plus the project workbench (fuzzy find-files, find-in-files, tree file ops, recent files, git). All UI chrome — menu bar, dialogs, file tree, command palette, toasts — comes from the shared zgui-core submodule (window.ZGui.*); the app hand-rolls no widgets. The GUI never edits files itself: every action is bridged into the editor by writing into the PTY (e.g. :open <path>).

~MODULE MAP

The Rust host (app/src-tauri/src) is seven small modules; the front end mounts the shared zgui baseline and bridges everything into the PTY. The zemacs editor itself is external — it runs inside the terminal, not as host code.

ModuleLayerRole
main.rsRust hostTauri entry point — registers terminal + GUI commands, wires the PTY's output/exit to webview events, sets up single-instance, deep-link and opener plugins.
terminal.rsRust hostEmbedded PTY adapter over the shared zpwr-embed-terminal crate; the editor runs inside this PTY, forwarding on_output/on_exit to the webview.
sidecar.rsRust hostResolves the bundled zemacs / stryke sidecars (env override → beside the executable → PATH) so the app is self-contained and never depends on PATH.
open_intake.rsRust hostFile-open intake — the MacVim "open from Finder / mvim CLI / mvim:// URL" surface; queues + live-emits open-files. Carries the unit tests.
fs_ops.rsRust hostRead-only directory listing backing the zgui file-open dialog (ZGui.modal + ZGui.tree); the open itself happens by writing into the PTY.
window_ops.rsRust hostWindow chrome the zgui menubar drives — MacVim fullscreen / transparency / blur analogs (no native UI).
project.rsRust hostThe project workbench backend: fuzzy find-files, find-in-files (regex/case/word), tree file ops (create/rename/delete/copy), persisted recent files, file stats and git status/branch/diff. Carries its own unit tests.
frontend/main.jsFrontendShell entry — mounts ZGui.appShell baseline and execs zemacs in the fullscreen embedded terminal.
frontend/menu.jsFrontendMacVim-style menu bar, ⌘-shortcuts and Open/Save-As/Help dialogs, built entirely from zgui-core widgets and bridged to the PTY.
frontend/panels.jsFrontendThe project workbench overlays (quick-open, find-in-files, recent, project files, git), added to the ⌘K palette; drives project.rs and opens results via the PTY.
frontend/terminal.jsFrontendShared zpwr-embed-terminal xterm.js frontend (PTY transport, geometry persistence).
frontend/i18n.jsFrontendShared zpwr-i18n runtime (window.t), loading the 27 JSON locale catalogs.
lib/zgui-coreSubmoduleShared cyberpunk UI baseline — ZGui.appShell (⌘K palette, colour-scheme picker, settings, CRT/splash), modal, tree, menubar, toast. Sole source of UI elements.

~DEPENDENCY FOOTPRINT

A deliberately lean dependency set: the Tauri runtime, three Tauri plugins for MacVim-style behaviour, and the shared embedded-terminal crate. Six git submodules are vendored, and the zemacs/stryke binaries ship as Tauri externalBin sidecars — no PATH dependency.

Six Vendored Submodules

frontend/lib/zgui-core (UI baseline), crates/zpwr-embed-terminal (PTY), crates/zemacs (the editor itself, a Helix fork), crates/zpwr-file-browser, crates/zpwr-i18n, and crates/zemacs-gui-core (shared GUI core + webui).

Self-Contained Sidecars

The zemacs and stryke binaries are bundled as Tauri externalBin sidecars, so the app has no PATH dependency.

Project Workbench

An IDE layer in the ⌘K palette (and shortcuts): Quick Open fuzzy finder (⌘P), Find in Files with regex/case/word (⇧⌘J), persisted Recent Files (⌘E), a Project Files tree manager (new/rename/delete/copy/info, ⇧⌘E) and a Git Changes panel (status + per-file diff). Results open at path:line:col in the editor.

~VERIFICATION

pnpm test (scripts/test.sh) runs two suites — a headless JS bridge suite and a Rust unit suite. The runner is honest about what it can build offline, and no end-to-end / UI automation exists yet.

SuiteCasesWhat it guards
crates/zemacs-gui-core/webui/menu.test.cjs32JS bridge tests — load menu.js in a vm context with a stubbed __TAURI__ and assert the exact escape sequences the menu actions write into the PTY, guarding the command mapping (zemacs is a Helix fork, so buffers, not tabs).
open_intake.rs5Rust #[test] cases covering the file-open intake parsing.
e2e / UI automation0None yet — the app is an early-stage MVP.

Honest Sidecar Skip

Because the Rust binary embeds tauri::generate_context!(), which validates the externalBin sidecars at compile time, the runner stages the sidecars first and honestly skips the Rust pass (rather than failing spuriously) when they can't be produced — e.g. offline with no zemacs build.

Early-Stage MVP

No end-to-end / UI automation exists yet. The editor runs in the embedded terminal with the zgui baseline; native tabs, menu bar and open/save dialogs are on the roadmap.
